# LLM Wiki 架构规范 (Schema)

这是一个遵循 LLM Wiki 模式的个人知识库。

在这个模式下，你作为LLM负责编写并维护 Wiki，而用户负责筛选来源、提出问题并指导分析工作。

## 目录结构 (Architecture)

```
my_wiki/
├── CLAUDE.md          # 本文件 — 架构与规范（LLM 需首先阅读此文件）
├── index.md           # 内容目录 — 列出所有 Wiki 页面及其摘要
├── log.md             # 变更日志 — 按时间顺序记录所有操作
├── raw/               # 原始来源（不可变，LLM 只读不写）
│   ├── articles/      # 文章、博客、gist
│   ├── papers/        # 学术论文
│   ├── assets/        # 图片与附件
│   └── notes/         # 个人笔记、会议记录
├── wiki/              # LLM 生成的 Markdown 页面（完全由 LLM 掌管）
│   ├── summaries/     # 来源摘要页（每个输入源对应一页）
│   ├── entities/      # 实体页（人物、组织、工具、产品）
│   ├── concepts/      # 概念页（想法、模式、技术）
│   ├── comparisons/   # 对比与分析页
│   └── meta/          # 综述、合成及元分析页面
├── templates/         # 页面模板，用于保持格式一致
└── LLM Wiki.md        # 原始模式文档（参考用）
```

## 规范 (Conventions)

### 页面格式

每个 Wiki 页面必须使用 YAML Frontmatter 包含以下字段：`title`（标题）, `type`（类型）, `tags`（标签）, `sources`（来源）, `created`（创建时间）, `updated`（更新时间）。

- **内部链接**：使用 `[[双链]]` 在 Wiki 页面之间跳转。
- **编辑备注**：使用 `> [!note]` 标注编辑评论或标记。
- **争议处理**：使用 `> [!warning]` 标注矛盾之处或未经验证的陈述。
- **标签规范**：使用小写并以连字符连接（kebab-case），例如：`#machine-learning`, `#personal-knowledge`。

### 命名规则

- **文件名**：全小写，使用连字符，不含空格（例如：`neural-networks.md`）。
- **摘要页**：以来源命名，不加前缀（例如：`llm-wiki-pattern.md`）。
- **实体页**：以实体名称命名（例如：`andrej-karpathy.md`）。

### 交叉引用

- 每个页面都应通过 `[[双链]]` 链接到相关页面。
- 更新页面时，检查是否需要添加指向其他页面或来自其他页面的新链接。
- 目标：消除孤立页面 —— 确保每个页面至少有一个入站链接。

## 工作流 (Workflows)

### 录入 (Ingest - 添加新来源)

1. 用户将源文件放入 `raw/` 对应子目录。
2. LLM 完整阅读该来源，与用户讨论关键要点。
3. 在 `wiki/summaries/` 中创建一个摘要页。
4. 针对提及的关键人物、工具、组织，在 `wiki/entities/` 中创建或更新实体页。
5. 针对关键想法，在 `wiki/concepts/` 中创建或更新概念页。
6. 更新 `index.md`：添加新页面并附上一句话摘要。
7. 在 `log.md` 中添加记录，格式为：`## [YYYY-MM-DD] ingest | 来源标题`。
8. 在日志条目中列出所有新建或更新的页面。

### 查询 (Query - 回答问题)

1. 阅读 `index.md` 以寻找相关页面。
2. 阅读相关的 Wiki 页面。
3. 综合答案，并使用 `[[双链]]` 进行引用。
4. 如果答案具有深度且可复用，将其归档为新的 Wiki 页面（存放在 `comparisons/` 或 `concepts/` 中）。
5. 在 `log.md` 中添加记录，格式为：`## [YYYY-MM-DD] query | 问题摘要`。

### 巡检 (Lint - 健康检查)

1. 扫描所有 Wiki 页面：查找孤立页面、缺失的交叉引用、陈旧的陈述以及矛盾之处。
2. 报告发现的问题，并直接修复简单明确的问题。
3. 建议值得进一步研究的新问题或需要寻找的新来源。
4. 在 `log.md` 中添加记录，格式为：`## [YYYY-MM-DD] lint | 健康检查`。

## 准则 (Guidelines)

- **宁多勿缺**：有疑问时就创建页面。页面的成本很低，而知识的遗失代价高昂。
- **优先更新**：与其创建重复页面，不如更新现有页面。
- **明确标注**：显式地标记矛盾之处，不要在更新时悄悄覆盖。
- **简洁全面**：摘要需简洁但信息完整，包含页码或章节引用。
- **独立可用**：即使 LLM 不在场，人类也应能在 Obsidian 中顺畅浏览此 Wiki。
